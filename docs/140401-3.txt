**1. Friends / DM Inbox Ordering**

In [friends/list.php](/C:/Users/REDKING/Projects/Chatapp/chatapp-core/friends/list.php:8), the backend used to return friends ordered by `friendships.created_at DESC`, which effectively meant “most recently added friend first.” That makes the rail feel like a contact list, not an inbox.

I changed the query so it now:
- `LEFT JOIN`s `dm_conversations c ON c.id = f.conversation_id`
- selects `c.updated_at AS conversation_updated_at`
- orders by `COALESCE(c.updated_at, f.responded_at, f.created_at) DESC`

That means the server now prefers:
- most recently active DM conversation first
- then most recently responded friendship
- then original friendship creation time

So even before the frontend does anything, the API is now closer to “recent conversations first.”

I also added a frontend sort in [FriendsRail.jsx](/C:/Users/REDKING/Projects/Chatapp/Application/src/features/friends/components/FriendsRail.jsx:6) so the UI reacts immediately to recent message activity using the already-loaded preview timestamps. I added:
- `getFriendActivityTimestamp(friend, conversationPreviews)`
- a `useMemo(...)` around `friendsWithTags`
- sorting by latest preview activity first, then username as a stable fallback

This means if conversation previews refresh locally, the rail can re-order right away without waiting for a full backend reload.

**3. Per-Message Delivery State**

You pointed out that users want better confidence around message delivery. There was already a relay/offline error path in [FriendsHome.jsx](/C:/Users/REDKING/Projects/Chatapp/Application/src/features/friends/pages/FriendsHome.jsx:1110), but it only showed a general error, not a status on the message itself.

I added a message-level delivery state flow.

In [friends/actions.js](/C:/Users/REDKING/Projects/Chatapp/Application/src/features/friends/actions.js:386), I changed `sendFriendDirectMessage(...)` so it captures the result of `sendDirectMessage(...)` instead of ignoring it. Then I return:

That gives the page layer a stable way to associate send-state with the actual outgoing message row.

In [FriendsHome.jsx](/C:/Users/REDKING/Projects/Chatapp/Application/src/features/friends/pages/FriendsHome.jsx:122), I added:
- `const [messageDeliveryById, setMessageDeliveryById] = useState({});`

Then I wired it in like this:

- on user change, it resets so old send-state does not leak between accounts
- in the existing `secureDmRelayQueueState` event handler, I now read:
  - `messageId`
  - `offlineRecipients`
  - `droppedRecipients`

Based on that event, I assign one of:
- `"sent"`
- `"queued"`
- `"failed"`

Specifically:
- if there are dropped recipients, it becomes `failed`
- else if there are offline recipients, it becomes `queued`
- otherwise it becomes `sent`

I kept the old relay-disabled error message too, so the global error still appears when needed.

Then after a successful `handleSendMessage(...)`, once the refreshed message list comes back, I mark the returned `outboundMessageId` as `"sent"` by default if it has not already been upgraded by the realtime queue event. That keeps the message from appearing untracked.

Finally I pass `messageDeliveryById` into the DM panel component so the thread renderer can use it.

**What I Did Not Claim / Change**

I did not add true end-to-end delivery receipts from the recipient device.

So the new status is honest about what the app actually knows today:
- `Sent securely` means the local send path completed and was not reported as dropped
- `Offline relay` means the realtime server explicitly reported offline recipients and queued the message
- `Failed` means the server said the message could not be queued because relay was disabled

It does not yet mean:
- the other person has opened the app
- the other device has decrypted the message
- the other user has seen it

That would need explicit receipt events.
