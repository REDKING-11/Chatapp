  - deprecated timestamp

**Reusable Share Transport And Security**
- Kept the existing encrypted DM chunk-transfer transport instead of turning reusable shares into a raw static-link system.
- Reusable shares do not reuse the same encrypted transfer payload forever.
- Each actual download request creates a fresh outgoing transfer session with a new transfer id and a fresh per-download encryption secret.
- The DM message can carry a stable `shareId`, but the actual file delivery still uses the current `dm:file:request`, `dm:file:ready`, `dm:file:chunk`, and `dm:file:complete` realtime flow.
- Before serving a reusable share, the sender now revalidates the local file and auto-rotates/deprecates the share if the file changed or disappeared.
- This keeps the reusable share identity stable from the UI/user perspective while avoiding a forever-reused static transfer secret behind the scenes.
