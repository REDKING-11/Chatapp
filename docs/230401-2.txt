**Slow Reliable Large File Transfers**
- Changed DM file delivery away from "push chunks as fast as possible" and toward an ack-driven transfer flow that advances one encrypted chunk at a time.
- Kept the existing `64 KiB` encrypted chunk size instead of trying to "fix" the problem by raising websocket payload or buffer limits.
- Added chunk acknowledgement behavior so the sender waits for the receiver to confirm each chunk before reading and sending the next one.
- Added retry behavior for missing acks, with bounded retries instead of infinite resend loops.
- Added shared sender throttling so one device serving multiple downloads does not hammer the realtime socket and trip buffer/rate limits.
- Made the receiver idempotent by offset so duplicate retried chunks are acknowledged but not appended twice.
- Added out-of-order offset rejection so a corrupted or skipped write does not silently poison the partial file on disk.
- Added written-byte / expected-byte tracking so completion is only accepted when the final file size matches what the sender announced.
- Kept the server lightweight: the realtime service still forwards `dm:file:*` signals without buffering file contents or persisting chunks.

**Behavior / security assumptions kept in this chat**
- Verified email is required before email-based password reset is trusted.
- Recovery keys are the required fallback path for all accounts going forward.
- No manual admin recovery path was added.
- Accounts that are already locked out and have none of:
  - verified email
  - MFA
  - previously generated recovery keys
  remain unrecoverable in this implementation.
- Password reset does not auto-login the user afterward.
