**Why this fixes the issue**
Before the patch:
- The app generated a bearer token.
- It inserted that exact token into `sessions.token`.
- Later it used that exact token in `SELECT` and `UPDATE` queries.
- Anyone with DB access could replay active sessions directly from the `sessions` table.

After the patch:
- The client still receives the bearer token and presents it normally.
- The server hashes that token before storing or querying it.
- The DB only needs the SHA-256 digest for lookup.
- Existing raw values are rewritten to a marker, so leaked DB rows no longer expose replayable bearer tokens.

That reduces the blast radius of a DB disclosure or admin misuse because the session table no longer contains the actual bearer secret in usable form.

**Compatibility and migration behavior**
I deliberately made this an in-place compatible migration:
- New installs or upgraded installs get `token_hash` automatically during runtime bootstrap.
- Existing sessions are migrated by hashing the old `token` value into `token_hash`.
- Once hashed, the old `token` column value is replaced with `sha256:<hash>` so it is no longer usable as a bearer token.
- Code continues to function even if the schema upgrade cannot run yet, because it falls back to the legacy behavior in that case.

That means the patch should not immediately break older environments, but the security improvement takes effect once the runtime schema migration succeeds.
