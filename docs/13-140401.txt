**3. DM / E2EE security work**
I implemented or tightened a lot of the security items you listed:

- Kept DM private keys inside the Electron main process and blocked secret-shaped IPC responses.
- Hardened the preload/main boundary so renderer code cannot directly access DM private keys.
- Kept DM store material in encrypted main-process storage guarded by `safeStorage`.
- Added runtime guards to prevent plaintext/key material from leaking through DM export paths.
- Enforced HTTPS/WSS for remote DM-related endpoints, allowing plain HTTP only for localhost development.
- Added signed DM device bundles and verification of those bundles on use.
- Added device revocation flow end to end, including UI and backend revocation endpoints.
- Encrypted DM attachments before transfer and put the attachment key inside the encrypted DM envelope.
- Ensured DM edits/deletes/reactions/disappearing-message events go through encrypted control messages instead of plaintext-style routes.
- Added a self-hosted separate ciphertext-only DM relay store instead of using plaintext channel message storage.
- Added device fingerprints / safety numbers and user-facing verification UI.
- Added trusted-device UX in settings so users can see active/new/revoked devices.
- Added Ed25519 signatures for DM message envelopes and verification on receipt.
- Added replay protection for DM messages.
- Added device key rotation and conversation rekeying.
- Added trusted-device approval for new DM devices.
- Reduced DM relay metadata retention and shortened retention choices.
- Moved long-lived auth token persistence out of renderer `localStorage` and into safer main-process storage.
- Added MFA/TOTP plus session management to account auth.
